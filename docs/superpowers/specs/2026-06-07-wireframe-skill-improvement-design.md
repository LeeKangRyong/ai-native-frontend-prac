## 1. 화면 계획 확정 단계 (Step 2 교체)

### 현재
xlsx의 `기능명` + `상위 기능 ID`만 보고 플로우를 묶어 간단 요약 출력 후 바로 그리기로 진행.

### 문제
- 기능명세서의 feature 이름이 실제 화면 타이틀과 일치하지 않는 경우 있음 (예: "하단 탭바", "상단 네비게이션바" 등이 raw feature 이름 그대로 화면 타이틀이 됨)
- 주요 노출 요소를 사전에 정의하지 않아 "스크롤 없이 한 화면에 다 보이게" 원칙을 지키기 어려움
- 사용자가 그리기 전에 화면 구성을 검토·수정할 기회 없음

### 변경 내용

**Step 2를 "화면 계획 확정" 단계로 교체한다.**

xlsx 전체 컬럼(기능 상세설명 포함)을 분석해 다음을 수행:
1. 각 feature가 독립 화면인지 판별 (기능 상세설명, 상위 기능 ID 기반)
2. 사용자 친화적인 화면 타이틀 도출 (raw feature 이름 그대로 쓰지 않음)
3. 각 화면의 주요 노출 요소 목록 도출 (스크롤 없이 보여야 할 요소)
4. 플로우별 테이블로 출력 → 사용자 수정/확정 대기

**출력 형식:**

```
## [TARGET] 화면 구성 계획 (우선순위: [PRIORITY])

총 N개 화면 / M개 플로우

### Flow 1: [플로우명]
| # | 화면 타이틀              | 주요 노출 요소                                    |
|---|--------------------------|---------------------------------------------------|
| 1 | [WF] 로그인              | 로고, 전화번호 입력, 로그인 버튼                  |
| 2 | [WF] 회원가입            | 이름·전화번호 입력, 약관 동의, 가입 버튼          |

### Flow 2: [플로우명]
| # | 화면 타이틀              | 주요 노출 요소                                    |
|---|--------------------------|---------------------------------------------------|
| 3 | [WF] 상단 네비게이션바   | 뒤로가기, 화면 타이틀, 알림 아이콘                |
| 4 | [WF] 하단 탭바           | 탭 5개 (홈/호출/이용내역/혜택/마이)               |

수정하고 싶은 타이틀이나 요소가 있으면 알려주세요.
(예: "3번 타이틀을 '네비게이션 컴포넌트'로", "1번에 '자동로그인 토글' 추가")
확정이면 "확정"이라고 해주세요.
```

**규칙:**
- "확정" 전까지 Figma API 호출 없음
- 사용자가 타이틀/요소를 수정하면 테이블 업데이트 후 재출력
- 확정된 화면 타이틀이 이후 Step 4의 `[WF] {화면 타이틀}` 프레임명으로 사용됨
- 확정된 주요 노출 요소가 Step 4 드로잉 시 "이 요소들을 스크롤 없이 한 화면에 배치" 기준으로 사용됨

## 2. Safe Area 강화

### 현재 문제
Safe Area 규칙이 스킬 내에 텍스트로만 기술되어 있어, Claude가 `use_figma` 코드 생성 시 실제로 위반하는 경우 발생.

### 변경 내용

**Step 4 드로잉 코드 상단에 픽셀 상수 블록을 고정한다.**

```javascript
// ⚠️ SAFE AREA CONSTANTS — 이 값을 절대 변경하지 않는다 (Mobile only)
const SA = {
  CONTENT_TOP: 59,    // StatusBar 하단 — 콘텐츠 최상단
  CONTENT_BOT: 810,   // HomeIndicator 상단 — 콘텐츠 최하단
  HEADER_Y: 59,       // NavBar 시작 y
  HEADER_H: 44,       // NavBar 높이
  CONTENT_START: 103, // 헤더 있는 화면의 콘텐츠 시작 y (59+44)
  TABBAR_Y: 776,      // TabBar 시작 y
  TABBAR_H: 34,       // TabBar 높이 (776+34=810)
  FAB_MAX_Y: 776,     // FAB/Floating 요소 최대 y
};
```

**요소 배치 검증 규칙 (코드 생성 시 반드시 적용):**
- 모든 요소: `y >= SA.CONTENT_TOP` AND `y + height <= SA.CONTENT_BOT`
- 위반 시 → y를 경계값으로 클램핑 후 배치

**화면 유형별 배치 기준 테이블 (Step 4에 포함):**

| 화면 유형 | 콘텐츠 시작 y | 콘텐츠 끝 y | 비고 |
|---|---|---|---|
| 인증/온보딩 | 59 | 810 | NavBar·TabBar 없음 |
| 메인 (TabBar 있음) | 103 | 776 | NavBar(59) + TabBar(776) |
| 서브 페이지 | 103 | 810 | NavBar(59), TabBar 없음 |
| 컴포넌트 화면 | 59 | 810 | 해당 컴포넌트만 표시 |

## 3. 섹션 크기 2x

### 현재 문제
화면 수 N에 딱 맞게 섹션을 생성해, 이후 `figma-add-wireframe`으로 화면을 추가할 때 섹션을 다시 늘려야 하는 번거로움 발생.

### 변경 내용

**섹션 가로 너비를 현재 화면 수의 2배로 pre-allocate한다.**

```javascript
// 현재
SECTION_W = padding*2 + maxScreensPerFlow * (frameW + frameGap)

// 변경 후
SECTION_W = padding*2 + (maxScreensPerFlow * 2) * (frameW + frameGap)
```

세로 높이는 현재 방식 유지 (플로우 행 수 기반).
